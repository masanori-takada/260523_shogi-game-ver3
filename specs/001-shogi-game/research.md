# Research: 将棋ゲーム

**Branch**: `001-shogi-game` | **Phase**: 0 | **Date**: 2026-05-23

## 1. プラットフォーム選定

**Decision**: Webブラウザ（クライアントサイド SPA）

**Rationale**:
- インストール不要でどのデバイスからもアクセス可能
- 将棋の盤面描画に Canvas/SVG/DOM どれでも対応可能
- 人間対人間モードで同一デバイスを共有して使いやすい
- 静的ファイルとしてデプロイ可能（サーバー不要）

**Alternatives considered**:
- デスクトップアプリ (Electron/Tauri): オーバースペック。配布コストが高い
- CLI: 将棋盤の視覚表現が困難。ユーザー体験が著しく低下
- モバイルアプリ: ネイティブ開発コストが高い。v1スコープ外

---

## 2. 言語・フレームワーク選定

**Decision**: TypeScript 5.x + Vite 5.x（UIフレームワークなし）

**Rationale**:
- TypeScript: 駒の種類・盤面座標・手の種別など列挙型が多く、型安全性がバグ防止に直結する
- Vite: 設定最小で TypeScript をネイティブサポート。開発サーバーが高速
- UIフレームワークなし: 将棋盤の描画は DOM 操作で十分。React/Vue の複雑さは不要

**Alternatives considered**:
- JavaScript (vanilla): 型なしによる将棋ロジックのバグリスクが高い
- React + TypeScript: コンポーネントモデルのオーバーヘッドが本ゲームには不要
- Svelte: 学習コストのわりにメリットが限定的

---

## 3. テスト戦略

**Decision**: Vitest + jsdom

**Rationale**:
- Vitest は Vite と同じ設定ファイルを共有。設定のオーバーヘッドなし
- jsdom でブラウザ DOM 環境をシミュレートし、UI ロジックもテスト可能
- ゲームエンジンのロジック（合法手生成・詰み判定）は純粋関数なのでユニットテストが容易

**テスト分類**:
- **ユニットテスト** (`tests/unit/`): ゲームエンジン関数（合法手生成、詰み判定、禁じ手検知）
- **統合テスト** (`tests/integration/`): 実際の対局フロー（対局開始→手を指す→終局）

---

## 4. ゲームエンジン設計

**Decision**: イミュータブルな状態管理 + 純粋関数による合法手生成

**Rationale**:
- GameState をイミュータブルにすることで、棋譜管理・将来の Undo 機能が容易
- 合法手生成を純粋関数にすることで単体テストが容易
- αβ枝刈りでの探索でも局面コピーが明確

**盤面表現**:
```
Board: Array(9 × 9) の Piece | null
座標: { row: 0-8, col: 0-8 } ← 内部表現（表示は 1-9 に変換）
先手: row が大きい側（画面下）
後手: row が小さい側（画面上）
```

**駒の種類（PieceType）**:
```
PAWN(歩), LANCE(香), KNIGHT(桂), SILVER(銀), GOLD(金),
BISHOP(角), ROOK(飛), KING(玉)
成り: PROMOTED_PAWN(と), PROMOTED_LANCE(成香), PROMOTED_KNIGHT(成桂),
      PROMOTED_SILVER(成銀), PROMOTED_BISHOP(龍馬), PROMOTED_ROOK(龍王)
※金・玉は成りなし
```

## 5. AI エンジン設計

**Decision**: Minimax アルゴリズム + αβ枝刈り

**Rationale**:
- 将棋AIの最も基本的かつ確立された手法
- 外部ライブラリ不要で実装可能
- 探索深度で難易度を調整できる（初級: 2手読み、上級: 4手読み）
- αβ枝刈りにより探索空間を大幅削減（枝刈りなしの約1/10〜1/100）

**評価関数**:
```
score = Σ(自駒の駒価値) - Σ(相手駒の駒価値)
      + 位置ボーナス（駒の位置による加点）
      + 持ち駒ボーナス
```

**駒の価値（参考）**:
| 駒 | 価値 | 成り後 |
|----|------|--------|
| 歩 | 100 | 400 |
| 香 | 300 | 400 |
| 桂 | 350 | 450 |
| 銀 | 500 | 500 |
| 金 | 600 | — |
| 角 | 800 | 1100 |
| 飛 | 1000 | 1300 |

**難易度設定**:
- 初級: 深さ2、応手時間目標 < 1秒
- 上級: 深さ4、応手時間目標 < 5秒

## 6. 将棋ルール実装上の注意点

**合法手生成で考慮が必要な特殊ルール**:

1. **打ち歩詰め**: 歩を打って詰みになる場合は禁じ手
2. **二歩**: 同じ筋に自分の歩（成り前）が2枚になる打ちは禁じ手
3. **行き所のない駒**: 歩・香を最奥段、桂馬を最奥2段に打つ/動かすのは禁じ手
4. **自殺手**: 自玉が王手にさらされる手は禁じ手
5. **千日手**: 同一局面が4回出現したら引き分け
6. **連続王手の千日手**: 同一プレイヤーが連続で王手をかけ続ける千日手は反則負け（v1では通常千日手として扱う）

**成りのルール**:
- 相手陣（先手: row 0-2、後手: row 6-8）に入る/出る/通過する際に成れる
- 行き所のない駒は強制成り
- 金・玉は成れない

## 7. ストレージ設計

**Decision**: ゲーム状態はメモリ管理のみ。ユーザー設定は localStorage

**保存項目**:
- ゲーム中の局面・棋譜: メモリ（ページリロードでリセット）
- 難易度設定: localStorage（`shogi-difficulty`）
- テーマ設定（将来拡張用）: localStorage（`shogi-theme`）

**Rationale**: v1スコープでは対局の保存・再開は対象外。シンプルさを優先。
